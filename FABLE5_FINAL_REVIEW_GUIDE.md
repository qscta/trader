# Fable5 最终独立审查指南

## 任务

对本分支进行一次独立、全量、对抗式终审。不要把现有审查说明、测试名称或 Codex 结论当作事实；它们只能作为线索。结论必须由你亲自从源码、测试和可复现证据得出。

目标优先级固定为：**稳定、安全、纯净、精简**。

## 部署门禁级安全属性

重点寻找能够导致下列结果的真实路径：

1. 意外新开仓，尤其是维护期、删除/禁用品种、人工仓位存在、重复日检或进程重启时。
2. 非幂等开仓或止损请求被重复发送，形成重复仓位或重复止损单。
3. 系统创建的真实仓位成为无止损残仓，或平仓后残留止损误伤未来仓位。
4. 本地账本、交易所实仓和止损单之间失配后，系统仍继续自动猜测、平仓、翻转或补挂。
5. 账本损坏、落盘失败、旧备份或错误交易所状态被静默接受。
6. 部署、systemd、进程锁、定时任务或 Web 即时开仓绕过新开仓门禁。

必须覆盖这些具体场景：

- 开仓请求超时、回执畸形、部分成交、成交确认失败、止损创建失败和回滚失败。
- 手动在 OKX 开仓、同方向加仓、减仓、反向操作或撤掉止损。
- 有持仓时从品种池删除/禁用，确认只平不开且结束后不再重入。
- HYPE 等非 1:1 `contractSize` 品种的币数/张数换算。
- 旧止损撤销确认、普通挂单与算法单清扫、重复/额外算法单隔离。
- 日线缺口、陈旧 K 线、未收盘 K 线、重复调度、启动补跑及 T+1 重入。
- `trade_state.json` 主文件损坏、备份存在、原子写失败及运行时补偿。
- API 即时开仓、手动平仓、参数修改、品种删除与日检/巡检并发。
- 资源监控日志路径与加固后的 `trading-mem-monitor.service`。
- `ops/sbin/trading-state-backup` 与当前应用 runner lock 的一致性。

## 明确的设计决定

- 前 N 日收盘价最高/最低值止损是双均线策略的一部分，不得当作海龟代码删除。
- 人工仓位或人工加减仓造成不一致时，正确行为是隔离、禁止自动管理并通知人工；不要建议自动认领或猜测财务归因。
- 删除/禁用品种的正确语义是“当前仓只平不开”。
- 证据不足时允许停机、拒单或人工裁决；不要求用更多状态机覆盖所有事故。

## 纯净与精简检查

1. 搜索并证明不存在海龟、ATR、Donchian、突破通道、金字塔加仓及双策略分派残留。
2. 区分运行时代码、测试、文档和历史兼容代码；不要用总行数直接判定臃肿。
3. 只有同时满足以下条件的精简建议才可提出：行为不变、安全属性不减、可被现有或新增测试证明，并明确净减少行数。
4. 不建议为结构品味把 mixin 改为组合，不新增状态类型，不增加猜测性自动恢复。

## 已知运维审查点

首轮审查确认了两个 P2：备份脚本的旧 `.runtime/runner.lock` 证明失真，以及资源监控缺 webhook 时以退出码 0 静默结束。当前分支包含候选修复：删除旧锁段（净减 11 行），并让生产监控路径退出码为 1、增加一项回归测试。请按 `FABLE5_FOLLOWUP_REVIEW.md` 独立复审；服务器尚未部署这些候选改动。

## 发现标准与输出格式

每个发现必须包含：

- 严重度（P0/P1/P2）；
- 精确文件和行号；
- 被破坏的安全属性；
- 可复现调用路径或最小失败测试；
- 为什么现有防线/测试没有挡住；
- 最小修复方案及其新增/删除行数估算。

纯推测、“理论上也许”、无法构造输入或调用路径的问题不得计为缺陷。P0/P1 阻止部署；已安全隔离且不影响交易正确性的 P2 记录但不扩张状态机。

最后分别给出：

1. 交易核心是否允许继续运行；
2. 海龟是否删除干净；
3. 是否存在值得现在实施的安全精简；
4. 备份运维链路应修复、删除还是改为人工隔离；
5. 最终结论及剩余风险。
