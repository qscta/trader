# 当前版本审查说明

本文件只描述当前欧易单所、双均线版本。历史版本的逐轮流水记录不随生产包部署，避免已经退出的功能继续污染代码检索、维护判断和交接上下文。

## 审查原则

- 稳定：不允许重复下单、无保护残仓、状态失忆或调度静默停止。
- 安全：证据不足时禁止猜测，隔离品种、保留保护并通知人工。
- 精简：只修开仓、订单、止损、状态和部署门禁问题；不增加新的恢复状态类型。
- 门禁：三轮独立全量对抗审查；发现问题的轮次不计干净轮；最终必须连续两轮无问题。

## 当前策略边界

- 唯一策略为 `ma_cross`。
- EMA 短、长周期决定方向。
- 开仓止损使用入场时前 N 根已收盘日线收盘价的最高值或最低值，持仓期间不自动推进。
- 从品种池删除或禁用但仍有持仓时，只管理到当前仓结束，不反手、不重入。
- 人工开仓不自动接管；人工加减仓导致交易所与本地数量不一致时，立即隔离并通知人工。

### 第一轮：交易与下单路径

### 第三轮：纯净性、精简度、前端与部署门禁

发现并修复：三个启动入口未共用单实例锁、维护期缺少统一开仓总闸、无生产调用的止损更新路径和一次性自动状态迁移仍在。已确认求索指数计算文件与图表库相对删除前基线完全一致，前端差异仅限退出功能的控件和文案。本轮有发现，不计干净轮。

### 第四轮：交易所证据与故障边界

发现并修复：日线时区与陈旧数据可进入真钱路径、空仓单次响应即可记平、止损创建未以待触发清单验证、超时成交数量归因不准、回滚失败可能遗忘残仓、主账本可能自动采用陈旧备份、持仓模式与调度线程未形成启动/存活硬门禁、非有限交易所数值可被误判为空仓或有效止损。所有修复均沿用现有持仓/残留模型，没有新增恢复状态类型。本轮有发现，不计干净轮。

### 第五轮：旁路写操作、清理竞态与告警证据

发现并修复：实盘验证脚本未与正式进程共用全机锁；非法/非有限平仓价可能污染账本；开仓前挂单清扫与无仓残留清理曾只信单次空仓响应；钉钉 HTTP 200 但响应无明确 `errcode=0` 时会误报推送成功。修复后全量 317 项测试、编译、Ruff、前端 JavaScript 语法与退出策略纯净性扫描均通过。本轮有发现，不计干净轮。

### 第六轮：跨文件崩溃窗口、告警送达与部署验证

发现并修复：交易所已平后本地记平与 T+1 记录之间存在崩溃窗口；孤儿仓首次钉钉发送失败后不会重试；固定 Gunicorn worker timeout 可能在订单/记账链中途杀进程；T+1 标记清除落盘失败时内存会与磁盘分叉；`verify_okx.py` 的默认保证金模式/杠杆会遮住真实配置，且新旧配置同时存在时读取优先级与生产系统不一致。修复仍只使用现有持仓、T+1、残留与人工隔离语义，没有新增恢复状态类型。全量 324 项测试、Python 编译、Ruff 未定义名/高风险 bugbear、前端 JavaScript 语法与退出策略纯净性扫描均通过。本轮有发现，不计干净轮。

### 第七轮：人工退出语义与部署门禁真实性

发现并修复：Web 手动平仓前未清除陈旧 T+1 标记，可能在后续日检中把用户明确退出的仓位自动补回；根 README 的快速启动命令遗漏 `--timeout 0`，与硬性部署要求自相矛盾；`verify_okx.py` 在取不到真实入场价时会跳过止损验证却仍可能汇总为通过，且实弹验证的开仓/挂止损失败被误归类为“非失败”。修复均为现有边界上的 fail-closed 收口，没有新增状态类型。全量 326 项测试、Python 编译、Ruff 未定义名/高风险 bugbear、前端 JavaScript 语法与退出策略纯净性扫描均通过。本轮有发现，不计干净轮。

### 第八轮：验证清理与自动化退出码

发现并修复：`verify_okx.py` 的实弹触发结果会在 `finally` 清理前提前求值，导致触发证据本来通过时，最终残仓或撤单不可确认仍无法推翻“通过”；命令行入口无论验证失败、未获得触发证据或凭据缺失都以退出码 0 结束，部署自动化可能把失败当成功；账户持仓模式检查结果被忽略，无法再次证明 `net_mode` 时仍会继续实弹验证。全部改为 fail-closed，且没有增加状态类型。全量 329 项测试、Python 编译、Ruff 未定义名/高风险 bugbear、前端 JavaScript 语法与退出策略纯净性扫描均通过；前端三文件哈希与审查前一致。本轮有发现，不计干净轮。

### 第九轮：验证工具自身的实盘残仓边界

发现并修复：`verify_okx.py` 把 `open_position` 返回失败直接当成绝对未成交，但适配层明确存在“请求结果不确定、交易所随后迟到成交”的孤儿仓窗口，验证工具会不清理就退出；普通与实弹验证均先撤止损再平仓，若平仓失败会主动把残仓变成无保护仓；最终清理只信一次空仓响应，弱于生产路径的连续空仓证明；只读验证拿不到有效 USDT 余额仍会退出 0。现统一为：开仓前连续证明空仓并确认挂单清空；一旦尝试开仓，无论调用返回成功与否都先 reduce-only 平仓、连续证明空仓，再验证式撤单；任一证据不足均非零失败，不新增状态。全量 332 项测试、三组各 332 项随机顺序测试、Python 编译、Ruff、前端语法与纯净性扫描均通过；前端哈希未变。本轮有发现，不计干净轮。

### 第十五轮：生产同构启动与合约张数换算

首次在服务器禁止新开仓门禁下启动候选版时，实盘只读同步稳定复现 HYPE 本地 10.1 币、交易所
101 张被误算为预期 100 张：二进制浮点 `10.1 / 0.1` 略低于 101，随后被交易所精度函数截断。
继续对换算做百万组攻击后，又复现“张数 × 极小合约面值”先产生略低币数的上游乘法残差。
现仅在币数反除前向正方向移动一个浮点 ULP，再以 Decimal 十进制相除并交给 ccxt 按真实步长
截断；真实不足下一步长的数量仍不进位。新增 10.1/0.1、真实不足一步长、极小面值乘法残差三项
永久回归。本轮有发现，不计净轮，服务器继续保持禁止新开仓。

### 第十六轮：开仓回执畸形与未分类提交异常

按不确定写操作执行既有撤单/回滚链。仅复用现有路径，不新增状态字段或恢复状态机。本轮有发现，

## 部署前必须人工确认

- 服务器实际持仓、算法单、普通挂单与本地账本一致。
- 状态文件位于项目根目录；任何旧目录中的状态都只允许人工核对、备份和迁移。
- 部署前先停止旧进程；新进程首次启动带 `TRADING_DISABLE_NEW_OPENS=1`。
- 确认 Web、调度、钉钉、止损巡检和账本正常后，才可解除开仓总闸。
- 依赖版本必须使用服务器已验证环境的锁定清单，不在部署时浮动升级。
