## 一、上实盘前必做（红线，代码审查替代不了）

- [ ] **OKX 账户为「单向(净)持仓模式」**——系统启动和 `verify_okx.py` 都会读取账户配置验证 `posMode=net_mode`；无法证明即拒绝启动。
- [ ] **模拟盘全链路验证**：`OKX_DEMO=1 python verify_okx.py BTCUSDT 0.01`
  逐项人工核对：
  - [ ] 张数换算符合预期（合约面值 contractSize）；
  - [ ] 止损为 reduce-only 条件单（多单 sell / 空单 buy）、触发后市价平仓、**只减不反向**；
  - [ ] `cancel_order` / 算法单查询能把止损撤干净（撤后「算法单应 0」）；
  - [ ] **止损单数量始终为 1**——若日志出现「算法单 2 个」，即超时重试产生了重复单，需排查；
  - [ ] 人工在 OKX 撤掉止损 → 等一个巡检周期 → 确认自动补挂 + 钉钉通知。
- [ ] **依赖锁定**：`requirements.txt` 已固定本轮验证过的直接依赖；在上线机器安装后再用
  `pip freeze > requirements.lock` 固化完整传递依赖，部署保存该锁文件。任何升级都重新过门禁。

## 二、部署硬性要求

| 要求 | 原因 / 校验 |
|---|---|
| 服务器时区 `Asia/Shanghai`（UTC+8） | 日检 08:00 对齐 OKX 日线收盘；启动校验 UTC 偏移，不符 critical 告警一次 |
| `gunicorn -w 1`（单 worker） | 多 worker 重复初始化交易系统；`wsgi.py` 文件锁兜底，多余 worker 直接拒启 |
| **禁止 `gunicorn --preload`** | 预加载会在 master 导入并启动交易线程，fork 后 worker 不继承线程；使用上方无 preload 的固定启动命令 |
| `gunicorn --timeout 0`（**必须显式加**） | 本进程把 Web 与交易调度放在同一 worker；任何固定秒数都无法覆盖“逐持仓行情查询 × 网络重试 × 算法单四类查询”的真实上界。Gunicorn 26 中 `0` 明确表示禁用 worker 静默超时，避免 master 在订单/记账中途杀掉唯一 worker；客户端慢请求超时由外层 HTTPS 反代控制 |
| 环境变量 `FLASK_SECRET_KEY` | 缺失**拒绝启动** |
| 环境变量 `TRADING_LOGIN_PASSWORD` | 管理台登录（未设则只能用 API Token） |
| 环境变量 `TRADING_COOKIE_SECURE=1`（**仅 HTTPS 部署**） | 会话 cookie 加 Secure；内网纯 HTTP 部署**不要设**，否则登录态无法保持 |
| 临时环境变量 `TRADING_SKIP_STARTUP_CATCHUP_ONCE=1`（仅部署晚间重启时临时使用） | 把本次启动当天标记为已日检，避免已过 08:00 后重启立刻补跑日检；次日正常恢复。未到当日检查时间时标志自动失效（不会吞掉当天正点日检） |
| 临时环境变量 `TRADING_DISABLE_NEW_OPENS=1`（部署核对期必须设置） | 集中阻断自动、即时、重入和反手的新开仓；持仓巡检、止损补挂和平仓仍正常。核对完成后才解除 |
| 环境变量 `TRADING_PROXYFIX_X_FOR`（可选，默认 1） | 声明反代跳数供登录防爆破还原真实客户端 IP：0=无反代直连（不信 XFF）、1=单反代或 Cloudflare Tunnel、2=CDN→nginx 双层。非法值拒绝启动 |
| **公网访问须经 HTTPS 反向代理**直连本 gunicorn | 登录防爆破按 `X-Forwarded-For` 单跳还原真实 IP（`ProxyFix(x_for=1)`）；
  裸 HTTP 暴露公网会让 XFF 可伪造、绕过登录限流 |
| 凭据用环境变量或 `config.json`（后者被 `.gitignore` 排除） | `OKX_API_KEY/SECRET/PASSPHRASE`、`DINGTALK_WEBHOOK` |

## 三、当前版本的运维影响

| 修复 | 运维须知 |
|---|---|
| 盘中巡检间隔 ≥60 分钟曾致调度静默崩溃 | 现 ≥60 分钟走 interval 触发器；`stop_loss_scan_interval_minutes` 可放心配 [1,1440] |
| ma_cross 开仓腿失败无恢复 | 现失败自动记 T+1，次日按 EMA 方向重入，恢复「永远在市」 |
| **钉钉 `access_token` 曾可能进日志** | **建议轮换一次钉钉机器人 webhook**，作废历史 `trading.log` 里可能残留的旧 token |
| `/api/qiusuo_index_ohlc?days=` 超大值 500 | 已钳制，无需关注 |
| 登录密码 / API Token 比较 | 已改恒定时间 bytes 比较，支持非 ASCII，无运维动作 |

## 四、日常巡检（面板 / 钉钉 / 日志）

- [ ] **顶栏红色警示 pill**：
  - 「止损残留阻断」= 撤销未确认，该品种已停新开仓，等自动清理或人工核对 OKX 委托；
  - 「止损异常待人工」= 止损单内容与本地不符 / 自动补挂失败，**立即**人工核对。
- [ ] **钉钉告警**关注：账本损坏拒启、权益采样连续失败（≈15 分钟）、时区异常、
  孤儿仓（交易所有仓本地无记录）、止损残留 / 补挂失败。
- [ ] **日志红旗**（`/api/logs` 或 `trading.log`）：
  - `算法单 N 个` 且 N>1 → 疑似重复止损单；
  - `拒绝启动` / `force_runtime` / `残留` → 状态异常，需人工。
- [ ] 每周核对钉钉周报的「未创新高天数 / 回撤」与预期一致。

## 五、部署顺序（允许短暂停机）

1. 停止旧交易进程并确认进程锁已释放。
2. 备份配置和全部状态文件；如状态位于其它目录，人工核对后迁移，代码不会自动搬运。`.bak` 只作人工候选，主账本异常时禁止自动覆盖。
3. 以 `TRADING_DISABLE_NEW_OPENS=1` 启动新进程。
4. 只读核对实仓、唯一匹配的 reduce-only 止损单、账本、Web、钉钉和调度线程。
5. 核对无误后解除总闸；已过日检时间的同日重启同时使用一次性补跑跳过标志。

## 六、资金变动（入金 / 出金）

- [ ] 到账后到「资金管理」卡**填净变动金额**（入金正 / 出金负）再点同步——精确锚定，不受点击时机影响；
