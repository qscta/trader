## 一、上实盘前必做（红线，代码审查替代不了）

- [ ] **OKX 账户为「单向(净)持仓模式」**——本系统的硬前提（`verify_okx.py` 会检测 `posMode`）。
- [ ] **当前提交的模拟盘全链路验证**：
  - `OKX_DEMO=1 python verify_okx.py BTCUSDT 0.01`
  - `OKX_DEMO=1 python verify_okx.py BTCUSDT 0.01 --side long --fire`
  - `OKX_DEMO=1 python verify_okx.py BTCUSDT 0.01 --side short --fire`
  - `OKX_DEMO=1 python verify_okx.py BTCUSDT 0.01 --side long --stop-id-reuse`
  - [ ] 两个 `--fire` 都真实触发并返回通过；超时未触发是**不确定**，调参重试，不得当通过。
  - [ ] `--stop-id-reuse` 返回通过（✓ OKX 接受终态后复用同一 algoClOrdId）。此项验证确定性
    止损 ID 的幂等假设——常规 `--fire`/基础验证测不到该场景（每次都是全新四元组）。若返回 🔴
    拒绝：系统在该场景会大声失败（回滚/告警+隔离，不丢钱），但「撤销后按完全相同参数补挂
    止损」将需要人工介入——上线前请知悉并记入运维预案。
  逐项人工核对：
  - [ ] 张数换算符合预期（合约面值 contractSize）；
  - [ ] 止损为 reduce-only 条件单（多单 sell / 空单 buy）、触发后市价平仓、**只减不反向**；
  - [ ] `cancel_order` / `cancel_all_orders` 对普通单与算法单都能完整分页复验，撤后持仓仍为 0；
  - [ ] **稳定态只有 1 张完整止损**。止损推进/部分平仓缩量采用 make-before-break，可短暂同时看到新旧 2 张；若跨过复验期仍多张才是异常，应立即人工核对；
  - [ ] 人工在 OKX 撤掉止损 → 等一个巡检周期 → 确认自动补挂 + 钉钉通知。
- [ ] **依赖锁定**：部署必须执行 `pip install -r requirements.lock`，并确认 Python 3.12；
  `requirements.lock` 已由生产候选环境固化入库。升级 ccxt 等直接依赖后必须重新完成全量测试、
  模拟盘实弹验证并重建锁文件，禁止部署时临时解析宽泛版本。

## 二、部署硬性要求

| 要求 | 原因 / 校验 |
|---|---|
| 服务器时区 `Asia/Shanghai`（UTC+8） | 日检 08:00 对齐 OKX 日线收盘；启动校验 UTC 偏移，不符 critical 告警一次 |
| `gunicorn -c gunicorn.conf.py` | 固定单 worker + gthread + 120 秒请求超时 + 900 秒优雅退出；项目内 runner 锁阻止所有入口重复初始化 |
| 环境变量 `FLASK_SECRET_KEY` | 至少 32 字节随机值；缺失/过短均**拒绝启动**。用 `python3 -c 'import secrets; print(secrets.token_hex(32))'` 生成 |
| 环境变量 `TRADING_LOGIN_PASSWORD` | 管理台登录（未设则只能用 API Token） |
| 环境变量 `TRADING_COOKIE_SECURE=1`（**仅 HTTPS 部署**） | 会话 cookie 加 Secure；内网纯 HTTP 部署**不要设**，否则登录态无法保持 |
| 临时环境变量 `TRADING_SKIP_STARTUP_CATCHUP_ONCE=1`（仅部署晚间重启时临时使用） | 把本次启动当天标记为已日检，避免已过 08:00 后重启立刻补跑日检；次日正常恢复。未到当日检查时间时标志自动失效（不会吞掉当天正点日检） |
| 策略行情固定读取最新 300 根 | 日检不为指标分页；最新已收盘日 K 陈旧、历史大跨度断层或窗口不足时该品种 fail-closed。EMA 长周期最大 149、EMA 止损最大 298、海龟通道最大 297 |
| 环境变量 `TRADING_PROXYFIX_X_FOR`（可选，默认 0） | 声明可信反代跳数：0=直连且完全不信 XFF、1=单反代、2=双层。只有实际存在对应可信反代时才能设非零 |
| 环境变量 `TRADING_RUNNER_LOCK_FILE`（仅项目目录只读时） | 指向当前用户所有的专用 0700 目录内；直指 `/tmp/runner.lock` 或符号链接目录会拒绝启动 |
| **公网访问须经 HTTPS 反向代理**，Gunicorn 默认仅监听 127.0.0.1 | 一层可信反代部署显式设 `TRADING_PROXYFIX_X_FOR=1`；直连保持 0，防伪造 XFF 绕过登录锁定；
  裸 HTTP 暴露公网会让 XFF 可伪造、绕过登录限流 |
| 凭据用环境变量或 `config.json`（后者被 `.gitignore` 排除） | `OKX_API_KEY/SECRET/PASSPHRASE`、`DINGTALK_WEBHOOK` |

资源监控单独使用仓库内 `systemd/trading-mem-monitor.service`。部署时复制到
`/etc/systemd/system/`，并创建权限 `0600`、仅含 `DINGTALK_WEBHOOK=...` 的
`/etc/trading-mem-monitor.env`，再执行
`systemctl daemon-reload && systemctl enable --now trading-mem-monitor`。它每 5 分钟只读
检查内存/磁盘，阈值为 85%/90%，不参与开平仓。**最小权限**：服务使用 systemd
动态用户运行，只接收专用 webhook 环境文件；沙箱明确禁止读取 `config.json`，也不加载
含 OKX 密钥的 `/etc/trading.env`。**磁盘监控覆盖 `/` 与账本数据目录所在盘**（同盘自动
去重、取更满者告警），避免数据卷独立挂载时写满而漏报。首次部署可在交易服务用户下用
`.venv/bin/python mem_monitor.py --test` 验证钉钉；服务日志位于
`/var/log/trading-mem-monitor/mem_monitor.log`，按 5MB 轮转保留 3 个备份，总量约 20MB。

## 三、本轮（部署后四轮扫描）已修与运维影响

| 修复 | 运维须知 |
|---|---|
| 盘中巡检间隔 ≥60 分钟曾致调度静默崩溃 | 现 ≥60 分钟走 interval 触发器；`stop_loss_scan_interval_minutes` 可放心配 [1,1440] |
| ma_cross 开仓腿失败 | 不推进当根 K 线游标，由 08:01/当日 30 分钟兜底重试；跨日不追补旧交叉，止损导致的 T+1 另走专用状态 |
| **钉钉 `access_token` 曾可能进日志** | **建议轮换一次钉钉机器人 webhook**，作废历史 `trading.log` 里可能残留的旧 token |
| `/api/qiusuo_index_ohlc?days=` 超大值 500 | 已钳制，无需关注 |
| 登录密码 / API Token 比较 | 已改恒定时间 bytes 比较，支持非 ASCII，无运维动作 |

## 四、日常巡检（面板 / 钉钉 / 日志）

- [ ] **顶栏红色警示 pill**：
  - 「止损残留阻断」= 撤销未确认，该品种已停新开仓，等自动清理或人工核对 OKX 委托；
  - 「止损异常待人工」= 止损单内容与本地不符 / 自动补挂失败，**立即**人工核对。
- [ ] **钉钉告警**关注：账本损坏拒启、权益采样连续失败（≈15 分钟）、时区异常、
  孤儿仓（交易所有仓本地无记录）、止损残留 / 补挂失败。
- [ ] **日志红旗**（`/api/logs` 或 `trading.log`）：
  - `算法单 N 个` 且 N>1 持续跨过复验期 → 疑似重复止损单；
  - `拒绝启动` / `force_runtime` / `残留` / `quarantine` / `open intent` / `close intent` / `signal_execution pending` → 状态机未收口，需人工核对交易所与账本；主动平仓 intent 会由启动、日检和 5 分钟巡检用基础腿+r1/r2 自动恢复，持续 unresolved 才需人工介入；
  - runner 线程不存活、连续跳过日检/盘中巡检、或单实例锁持续冲突 → 视为调度不健康，不能只看 Web 页可打开。
- [ ] 每周核对钉钉周报的「未创新高天数 / 回撤」与预期一致。

## 五、资金变动（入金 / 出金）

- [ ] 到账后到「资金管理」卡**填净变动金额**（入金正 / 出金负）再点同步——精确锚定，不受点击时机影响；
      留空则须在到账后 5 分钟内点击（否则采样污染求索指数）。

## 六、归档、备份与灾难恢复

- [ ] `trade_state.json` 只保留最近 200 笔平仓；更旧记录已按平仓年份无损写入 `closed_trades_archive_YYYY.json`（旧 `closed_trades_archive.json` 继续只读兼容），`/api/trades` 仍统一分页。年度文件可独立备份，但不得手工删历史。
- [ ] 备份必须在停止 runner 后做**整体一致快照**：`config.json`、两个 owner/migration 点文件、`trade_state.json*`、`closed_trades_archive.json*`、权益/日线/ticks 主备文件及当时代码提交。备份含凭据，必须加密、限权，不得进 Git/聊天/工单。
- [ ] 恢复时不要只拷一个主账本：在隔离目录恢复整套同一时点文件，保留原目录离线副本，先只读核对 schema/owner/本地仓位/交易所仓位/挂单，全部一致后再启 runner。主备同坏时系统会拒启，不得用空文件强行绕过。
